## Description

Implement the LED driver module (`firmware/src/led_rmt.rs`) that
drives four 16Г—16 WS2812B panels via RMT DMA channels. Each panel has
256 LEDs chained serially. The driver includes a pattern engine that
generates frames from multiple sources and pushes them via DMA with no
CPU blocking.

## Pattern Engine Architecture

**effects** map (pixel_index, time, palette) вҶ’ RGB, and
**bitmaps** bypass the effect layer entirely for pre-rendered content.

### Tier 1: Static Presets (hardcoded, ~500 bytes)

Simple `enum` variants with no computation. Each preset fills the
frame buffer with a constant or repeating pattern.

```rust
enum StaticPreset {
    Solid(Rgb),             // single color, e.g. red
    GradientH(Rgb, Rgb),   // horizontal gradient
    GradientV(Rgb, Rgb),   // vertical gradient
    Checker(Rgb, Rgb, u8), // checkerboard, cell size
    Off,                    // all black
}
```

These are the only patterns that don't animate вҖ” they're "background
ambiance" for when nothing is playing.

### Tier 2: Procedural Effects (hardcoded, ~3-5KB total)

Each effect is a pure function `(x: u8, y: u8, t_ms: u32, palette: &[Rgb; 16]) -> Rgb`.
All math is integer/fixed-point вҖ” no float, no alloc.

| Effect        | Algorithm                                                    | Use Case                     |
|---------------|--------------------------------------------------------------|------------------------------|
| Fire          | 1D cellular automaton + palette LUT, hotвҶ’cold floorвҶ’ceiling  | "fireball" keyword           |
| Plasma        | Sum of sine waves with prime-number periods, palette-mapped  | "magic" / ambient            |
| Rainbow       | HSV hue sweep over time, fixed saturation/value               | Transition / celebration     |
| Flash         | Exponential decay from white to black, timed trigger          | "thunder" / "explosion"      |
| Ripple        | 2D sine wave radiating from center, palette-mapped            | Water / "healing"            |
| Sparkle       | Random pixels fade in/out, Poisson-like timing                | "fairy dust" / ambient       |

**Where "moods" come from**: A mood like "volcano landscape" or "lush
jungle biome" is a **procedural effect + a specific palette + tuned
parameters**.  For example:

- Volcano: Fire effect + `[black, dark_red, red, orange, yellow, white]` palette
- Lush jungle: Sparkle effect + `[dark_green, green, lime, light_green, yellow_green]` palette
- Ocean depths: Ripple effect + `[navy, blue, cyan, teal, white]` palette

The palette + effect combination is stored as a small config blob
on SD (or sent from the companion app). Users don't write code вҖ”
they select from presets and tune sliders.

### Tier 3: Bitmap Playback from SD (no code, just data)

For complex, hand-crafted animations that can't be expressed
algorithmically, the companion app uploads raw 16Г—16 RGB frames.
These are stored on SD as `.bin` files (768 bytes per frame) and
played back at a configurable frame rate.

```
/patterns/
  0.bin    вҶ’ raw frames [768B][768B]...  (keyframe animation)
  1.bin    вҶ’ single frame                 (static image)
  index.json вҶ’ { 0: {name: "volcano", fps: 12, frames: 32}, ... }
```

The firmware just DMAs the buffer from SD вҶ’ RMT. No decode needed.

### Tier 4: Companion App Pattern Editor (Post-MVP)

The Dioxus companion app includes a simple 16Г—16 pixel-art canvas:

- Draw pixels with a color picker
- Import an image вҶ’ auto-downscale to 16Г—16 with nearest-neighbor
- Chain frames into a keyframe animation (spritesheet-style)
- Preview on a simulated LED grid before uploading
- Export as `.cubepattern`

This is how users create custom "moods" without writing code.

## How the Industry Does It

| Product          | Approach                                                  |
|------------------|-----------------------------------------------------------|
| **WLED**         | ~50 effects (math functions) + ~40 palettes (color ramps). Ships with all effects in firmware. User picks effect + palette. |
| **FastLED**      | Library of effect functions. User code calls `fire2012(leds, palette)`. All procedural, zero pre-rendered content. |
| **Pixelblaze**   | On-device JS-like expression evaluator. Users write `hsv(x/16 + t, 1, 1)`. Ships with curated patterns. Patterns are text, not binaries. |
| **DMX consoles** | Keyframe-based pixel mapping. All content pre-rendered by a lighting designer. High bandwidth, no on-device generation. |
| **CubeMaster**   | Hybrid: hardcoded procedural effects + palette system + bitmap fallback for custom moods. |

**CubeMaster's advantage**: The palette+effect decoupling gives users
creative control without writing code or uploading large assets.
For power users, the bitmap path handles anything procedural can't.


8. Implement 30Hz tick with frame generation вҶ’ DMA push
9. Test with 1 panel first, then scale to 4

## Acceptance Criteria

- 4 panels refresh at 30 FPS without visible tearing
- Each procedural effect produces visually distinct, smooth animation
- Palette switching visibly changes effect appearance (Fire+OCEAN vs Fire+FIRE)
- Bitmap playback from SD achieves вүҘ12 FPS for 32-frame animation
- CPU usage <10% for LED DMA + pattern generation
- Adding a new procedural effect requires only: 1 function + 1 variant in `EffectId`
- Binary size <12KB for pattern engine (effects + palettes + engine)
