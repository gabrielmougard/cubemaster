To drive a speaker from an ESP32-S3, you need an **audio amplifier**.

<summary><strong>Approach 1: Using a dedicated audio amplifier module (Recommended)</strong></summary>

This is the most common, practical, and recommended method for a project like this. You'll use the ESP32-S3 to send an audio *signal* (either digital or analog depending on the amplifier) to an amplifier module, which then powers the speaker.

### How it works:
1.  **ESP32-S3:** Generates the audio data (e.g., from an audio file, streaming, or synthesized). This data needs to be sent to the amplifier.
2.  **Audio Amplifier Module:** Takes the low-level audio signal from the ESP32-S3 and amplifies it to a level capable of driving the speaker.
3.  **Speaker (Woofer or Tweeter):** Connects to the output of the amplifier module.

# Spatial Audio System with ESP32-S3

## Short Answer

- **You cannot drive speakers directly from the ESP32-S3.** Its GPIO outputs ~12 mA at 3.3 V (≈40 mW max) — about 500× too weak even for a tweeter. You need power amplifiers.
- **Best amp choice: 3× TAS5805M** (or TAS5825M) — I²S-input Class-D amplifiers with **built-in DSP**. They handle the crossover, EQ, and gain entirely in hardware after one-time I²C configuration at boot.
- **You do NOT need a separate audio processor.** The DSP inside the TAS58xx family replaces it.
- The ESP32-S3 streams 5 channels of PCM over **I²S in TDM-8 mode**; each amp picks its two slots.

---

│  ESP32-S3    │ ───LRCLK───► │ (PBTL mono)  │
│              │ ───DOUT────► └──────────────┘
│  + I²C0      │      │       ┌──────────────┐
│              │      ├──────►│ TAS5805M #2  │── Tweeter L1 (10W)

async fn audio_task(mut i2s_tx: I2sTx<'static, DMA>) {
    // Double-buffered DMA: while DMA plays buffer A, you fill B.

### Spatial panning math (purely static gain matrix)

Since you said **"not really alter the sound at runtime"**, you only need to compute per-tweeter gains from a virtual source position. The classic approach for 4 speakers on a square is **VBAP** or simple **equal-power panning**:

For a source at angle $\theta$ (azimuth around the cube), with tweeters at angles $\phi_i \in \{45°, 135°, 225°, 315°\}$:

$$g_i(\theta) = \max\!\left(0,\ \cos\!\left(\tfrac{\pi}{2} \cdot \tfrac{\theta - \phi_i}{45°}\right)\right)$$

then normalize so that $\sum_i g_i^2 = 1$ (equal-power).

```rust
fn pan_gains(theta_deg: f32) -> [f32; 4] {
    const PHI: [f32; 4] = [45.0, 135.0, 225.0, 315.0];
    let mut g = [0.0_f32; 4];
    let mut sum_sq = 0.0;
    for i in 0..4 {
        let d = ((theta_deg - PHI[i] + 540.0) % 360.0) - 180.0;
        let x = (d.abs() / 45.0).min(2.0);
        g[i] = ((core::f32::consts::FRAC_PI_2 * (1.0 - x/2.0)).cos()).max(0.0);
        sum_sq += g[i] * g[i];
    }
    let norm = 1.0 / sum_sq.sqrt();
    g.map(|v| v * norm)
}
```

### Per-frame rendering

```rust
fn render_frames(out: &mut [i16]) {
    let gains = pan_gains(current_source_angle());
    for frame in out.chunks_mut(8) {
        let s = decode_next_mono_sample();   // i16 from WAV/MP3/stream
        frame[0] = (s as f32 * gains[0]) as i16; // tweeter 1
        frame[1] = (s as f32 * gains[1]) as i16; // tweeter 2
        frame[2] = (s as f32 * gains[2]) as i16; // tweeter 3
        frame[3] = (s as f32 * gains[3]) as i16; // tweeter 4
        frame[4] = s;                            // woofer (raw — amp DSP filters)
        // frame[5..8] = unused
    }
}
```

**Important**: you push the *full-band* mono signal to every channel and let each amp's biquads carve out its band. This is much simpler than implementing FIR/IIR crossovers in Rust on the ESP32.

<details>
<summary><b>I²S TDM configuration details on ESP32-S3</b></summary>

- The S3's I²S peripheral supports TDM with up to 16 slots. Use `Standard::Tdm` mode in `esp-hal`.
- 8 slots × 16 bits × 48 kHz = 6.144 MHz BCLK — well within spec.
- Connect all 3 amps to the same BCLK / LRCLK / DOUT lines.
- Each amp's I²C config selects which two slots it consumes (registers `TDM_CFG2`/`TDM_CFG3`).
- Tie each amp's I²C address differently via the `ADR` pin straps so you can address them individually.

</details>

## What Changes in Your Rust/Embassy Code

Massively simpler than the 5-channel TDM version:

```rust
// Standard stereo I²S, 48 kHz, 16-bit — esp-hal default config works.
fn render_frame(out: &mut [i16; 2], source_angle_deg: f32) {
    let s = decode_next_mono_sample() as f32;
    // Equal-power stereo pan, -90°..+90° mapped to L..R
    let t = (source_angle_deg.clamp(-90.0, 90.0) + 90.0) / 180.0; // 0..1
    let l_gain = ((1.0 - t) * core::f32::consts::FRAC_PI_2).cos();
    let r_gain = (t * core::f32::consts::FRAC_PI_2).sin();
    out[0] = (s * l_gain) as i16;
    out[1] = (s * r_gain) as i16;
}
```

That's the entire spatial logic. The amp board (Path A) splits low frequencies to the woofer internally; you never think about crossovers in firmware.

- front tweeter louder when source is in front
- left tweeter louder when source pans left
- right tweeter louder when source pans right
- rear tweeter louder when source is behind

If you want more realism, add tiny delays:
