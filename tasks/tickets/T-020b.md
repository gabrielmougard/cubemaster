## Description

Build a Rust desktop tool (integrated into the companion app or
standalone) that converts a video file into a CubeMaster-compatible
playback format, mapping the video's visual content onto the four
lateral faces of the cube as if it were a 3D projection.

## Algorithm: Equirectangular → Cubemap

The core technique is the **equirectangular-to-cubemap** conversion.
Treat the source video as an equirectangular panorama (or apply a
panoramic interpretation to any video), then sample four 90°-FOV
views around the equator — one per cube face.

```
Source video frame        Cube lateral faces (4 × 16×16)
┌─────────────────────┐      ┌────┐ ┌────┐ ┌────┐ ┌────┐
│                     │      │ F0 │ │ F1 │ │ F2 │ │ F3 │
│    ┌─┐ ┌─┐ ┌─┐ ┌─┐ │      │    │ │    │ │    │ │    │
│    │F│ │F│ │F│ │F│ │  →   │ 16 │ │ 16 │ │ 16 │ │ 16 │
│    │0│ │1│ │2│ │3│ │      │ ×  │ │ ×  │ │ ×  │ │ ×  │
│    └─┘ └─┘ └─┘ └─┘ │      │ 16 │ │ 16 │ │ 16 │ │ 16 │
│                     │      └────┘ └────┘ └────┘ └────┘
└─────────────────────┘
```

For each pixel `(xf, yf)` on face `f ∈ {0,1,2,3}` at resolution 16×16:

1. Compute a 3D direction vector from the cube center through that
   face pixel (standard cubemap math — each face covers a 90° FOV
   quadrant of the unit sphere)

2. Convert the direction vector to equirectangular UV coordinates on
   the source video frame

3. Sample the source video at that UV, optionally with bilinear
   filtering for smoother downscaling

### Face-to-3D Mapping

Each lateral face is a plane tangent to the unit cube, centered at
a 90° azimuth offset:

| Face | Azimuth center | Normal vector   |
|------|---------------|-----------------|
| 0    | 0° (front)    | (0, 0, 1)       |
| 1    | 90° (right)   | (1, 0, 0)       |
| 2    | 180° (back)   | (0, 0, -1)      |
| 3    | 270° (left)   | (-1, 0, 0)      |

### Equirectangular UV ↔ 3D Direction

```
u_equi = 0.5 + atan2(dir.z, dir.x) / (2 * PI)   // longitude → horizontal
v_equi = 0.5 - asin(dir.y) / PI                   // latitude → vertical
```

These are textbook formulas used in every 360° video player and VR
rendering pipeline.

### FFmpeg One-Liner (Quick Prototype)

```bash
ffmpeg -i input.mp4 \
  -vf "fps=12,scale=64:64,v360=e:c3x2:cubic:out_pad=0,scale=16:16" \
  frame_%04d.png
```

This extracts 12 FPS × 4 faces in a 3×2 cubemap layout, then crops to
16×16. The top/bottom faces are discarded.

## Rust Implementation

A minimal Rust CLI using `ffmpeg` as a subprocess (or the `ffmpeg-next`
crate for in-process decoding):

```rust
// Pseudocode for the converter core
struct CubeConverter {
    face_res: u16,  // 16
}

impl CubeConverter {
    fn convert_frame(&self, frame: &[u8], width: u32, height: u32) -> [RgbFrame; 4] {
        let mut faces = [RgbFrame::new(); 4];
        for face in 0..4 {
            let azimuth = face as f32 * 90.0_f32.to_radians();
            for y in 0..self.face_res {
                for x in 0..self.face_res {
                    let dir = self.face_pixel_to_dir(face, x, y);
                    let (u, v) = dir_to_equirect(dir, width, height);
                    let color = sample_bilinear(frame, width, u, v);
                    faces[face].set(x, y, color);
                }
            }
        }
        faces
    }

    fn face_pixel_to_dir(&self, face: u8, x: u16, y: u16) -> Vec3 {
        // Map [0..16] to [-1..1] range, flipped Y
        let u = (x as f32 / 15.0) * 2.0 - 1.0;
        let v = (y as f32 / 15.0) * 2.0 - 1.0;
        // Each face is a 90° FOV slice around the equator
        let az = face as f32 * std::f32::consts::FRAC_PI_2;
        Vec3 { x: -u * az.cos(), y: -v, z: u * az.sin() }.normalize()
    }
}
```

The output is a `.cubepattern` file: a binary stream of `[R;256][G;256][B;256]`
per face per frame, with a small JSON header describing FPS and frame
count.

## Seamless Loop Detection

A looping pattern should play continuously without a visible jump when
it restarts. The tool automatically finds the optimal loop point.

### Algorithm

For a video of N frames, find the frame pair (i, j) with the smallest
perceptual difference that respects a minimum loop duration.

```
┌─────── frames ───────────────────────────┐
│ 0  1  2  ...  i ─► ... ─► j-1  j  ... N-1│
│                 ▲              │           │
│                 └── loop back ─┘           │
│                   (j blends into i)        │
└────────────────────────────────────────────┘
```

Steps:

1. Extract all N frames from the video (already have them from the
   conversion step)
2. Set `MIN_LOOP_FRAMES = 24` (2 seconds minimum at 12 FPS)
3. For each candidate start frame `i` from 0 to `N - MIN_LOOP_FRAMES`:
   - For each candidate end frame `j` from `i + MIN_LOOP_FRAMES` to `N`:
     - Compute `distance = frame_diff(frame[i], frame[j])`
     - Track best `(i, j)` with minimum distance
4. The loop region is `frames[i..j]` — these play forward, then jump
   back to frame `i`

### Frame Difference Metric (L1 SAD)

Sum of absolute differences across all 4 faces, all pixels, all RGB
channels:

```
distance = Σ(f=0..3) Σ(p=0..255) (|Rf[p]-Rj[p]| + |Gf[p]-Gj[p]| + |Bf[p]-Bj[p]|)
```

Range: 0 (identical frames) to 4 × 256 × 3 × 255 ≈ 783,360 (max diff).

### Crossfade at Loop Point

To hide the remaining discontinuity at the best match, crossfade the
last `CROSSFADE_FRAMES` (4 frames = 333ms) of the loop into the first
`CROSSFADE_FRAMES`. The output removes the last few frames and blends
them into the first:

```
Output: frames i, i+1, ... j-CROSSFADE-1
Loop action at end: blend last frame into frame i with alpha fade
```

### Result

For typical ambient video (slow-moving clouds, fire, water), the
algorithm finds a loop point with near-zero visual discontinuity.
For fast-action video, the user can manually override the loop region
in the preview UI.

## Compression: `.cubepattern` Format v2

Raw frames are 3072 bytes each. For longer patterns, compression
reduces SD usage and upload time. The format uses a simple intra-frame
**RLE** + inter-frame **delta** codec that requires minimal decoder
complexity on the ESP32.

### Bitstream Structure

```
┌──────────────────────────────────────────────────────────────┐
│ Header (16 bytes)                                            │
│  magic:       [4B] "CUBE"                                    │
│  version:     [1B] 0x02                                      │
│  fps:         [1B] 1–30                                      │
│  frame_count: [2B] u16 LE                                     │
│  flags:       [1B] bit0 = compressed (1)                     │
│  keyframe_int:[1B] keyframe every N frames (0 = all keyframe)│
│  loop_start:  [2B] frame index (from seamless loop detector) │
│  loop_end:    [2B] frame index                                │
│  reserved:    [2B]                                            │
├──────────────────────────────────────────────────────────────┤
│ Frame 0 (keyframe)                                           │
│  type:    [1B] 0x00 = keyframe                                │
│  data:    [3072B] raw RGB (RLE-compressed if flags&1)        │
├──────────────────────────────────────────────────────────────┤
│ Frame 1 (delta)                                              │
│  type:    [1B] 0x01 = delta                                   │
│  data:    [varies] delta-compressed payload                   │
├──────────────────────────────────────────────────────────────┤
│ ...                                                           │
└──────────────────────────────────────────────────────────────┘
```

### RLE Encoding (Keyframes)

Each keyframe is compressed with a byte-level RLE:

```
[run_length: u8][pixel_value: u8]
```

A run of 3 identical pixels `R,G,B` → `[3][R][G][B]`.
Non-repeating pixels are stored as literal runs:
- Run length 0–127: literal run of N+1 unique pixels
- Run length 128–255: repeat run of N-127 copies of the next 3 bytes

**Compression ratio**: For LED patterns (large solid areas), typical
2–4×. For photographic content, 1–1.5×.

### Delta Encoding (Inter-frames)

Each inter-frame stores only the pixels that changed from the previous
frame:

```
[skip_count: u8]    — pixels to skip (unchanged), 0–255
[change_count: u8]  — pixels to change (0 = end of frame)
[data: 3×change_count bytes] — new RGB values
... repeat until change_count = 0
```

**Compression ratio**: For slow ambient patterns (fire, water ripple),
10–50×. Many frames have <10% pixel changes. The decoder on ESP32
is a simple loop: skip N, copy M, repeat.

### Decoder Pseudocode (ESP32)

```rust
fn decode_frame(
    prev: &mut [u8; 3072],
    input: &[u8],
    output: &mut [u8; 3072],
) -> usize {
    let kind = input[0];
    let data = &input[1..];
    match kind {
        0x00 => {
            // Keyframe — RLE decode directly into output
            rle_decode(data, output);
            output.copy_from_slice(&output[..3072]); // also update prev
            data.len() + 1
        }
        0x01 => {
            // Delta — start from previous frame, patch changes
            output.copy_from_slice(&prev[..3072]);
            let mut pos = 0;
            let mut idx = 0;
            while idx < data.len() {
                let skip = data[idx] as usize; idx += 1;
                pos += skip * 3;
                let change = data[idx] as usize; idx += 1;
                if change == 0 { break; }
                output[pos..pos + change * 3]
                    .copy_from_slice(&data[idx..idx + change * 3]);
                idx += change * 3;
                pos += change * 3;
            }
            prev.copy_from_slice(&output[..3072]);
            idx + 1
        }
    }
}
```

~30 lines of decoder. No heap, no alloc, pure stack.

### Expected Compression

| Content type    | Raw (10s) | Compressed | Ratio |
|-----------------|-----------|------------|-------|
| Photographic video | 360 KB | ~120 KB | 3× |
| Ambient fire | 360 KB | ~50 KB | 7× |
| Slow color cycle | 360 KB | ~15 KB | 24× |
| Static mood | 360 KB | ~3 KB | 120× |

### Decision for MVP

Start with **keyframe-only RLE** (simplest decoder, ~15 lines). Add
delta encoding in v2 if needed. The 16GB SD card holds thousands of
uncompressed patterns anyway — compression is a nice-to-have, not
blocking.

## Performance

- Per frame: 4 faces x 256 pixels x (3D math + bilinear sample) = ~4K ops
- At 12 FPS: ~48K ops/second, trivial on desktop CPU
- Video decode is the bottleneck, handled by ffmpeg hardware decode
- Loop detector: O(N^2) frame comparisons, N ~ 120 for 10s at 12 FPS clip
  (14400 comparisons x 3KB = 42MB scanned, ~100ms on desktop)

## Acceptance Criteria

- 10s MP4 video converts to `.cubepattern` in under 30 seconds
- All 4 faces show content at correct 90 degree azimuth offsets
- Seamless at cube edges: face 3 right edge connects to face 0 left edge
- Seamless at loop point: playback cycles 5 times with no visible jump
- RLE + delta decode runs on ESP32 with under 2KB stack and under 5ms CPU per frame
- Compressed `.cubepattern` is at most 50% of raw size for ambient content
- Bilinear sampling prevents aliasing at 16x16 resolution
